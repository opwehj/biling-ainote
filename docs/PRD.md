## 1. 产品概述

### 1.1 产品名称候选

| 候选名称 | 含义 | 说明 |
|----------|------|------|
| **笔灵（BiLing）** | "笔" + "灵"（灵动/智能） | 简洁好记，突出智能笔记得灵动特性，推荐 |
| **智记（ZhiJi）** | "智"（智能）+ "记"（记录） | 直白表达智能记录的核心功能 |
| **灵犀笔记** | 取"心有灵犀"之意 | 寓意 App 能理解用户意图，稍长但有意境 |

### 1.2 产品定位

> **一句话定位**：一款基于国内大模型的多模态 AI 笔记应用，用户通过文本、图片、音频输入素材，AI 自动提取内容并生成结构化笔记。

### 1.3 目标用户

| 用户群体 | 典型场景 | 核心痛点 |
|----------|----------|----------|
| **学生** | 课堂拍照、录音，课后整理笔记 | 手动整理课堂内容耗时，OCR + 转写能大幅提效 |
| **职场人士** | 会议录音、白板拍照，会后生成纪要 | 会议纪要撰写繁琐，需要快速结构化输出 |
| **研究人员** | 文献拍照、访谈录音，提取关键信息 | 大量素材难以高效归纳，需要摘要与结构化 |
| **内容创作者** | 灵感速记、素材收集，辅助内容产出 | 零散素材需要快速整合为可用内容 |
| **记者/采访者** | 采访录音转写、现场拍照记录 | 需要快速将音频/图片转为可编辑的文字稿件 |

### 1.4 核心价值主张

1. **多模态输入，一站式笔记**：文本、图片、音频均可作为输入源，无需在多个 App 间切换
2. **AI 自动整理，告别手动排版**：大模型自动进行 OCR、语音转写、内容摘要与结构化，输出可直接使用的笔记
3. **国内大模型加持，访问稳定**：基于通义千问/智谱 GLM/文心一言等国内模型，网络稳定、响应快速
4. **隐私可控，API 自配**：用户可自行配置 API Key，数据不经第三方中转

## 4. 功能模块设计

### 4.1 输入模块

负责采集用户的多模态素材，是笔记生成的数据源。

#### 4.1.1 文本输入

- **功能**：提供多行文本输入框，支持直接键入或粘贴长文本
- **交互**：在"新建笔记"页面，文本输入为默认 tab，点击即可输入
- **约束**：单次输入文本上限 10,000 字符；支持自动检测输入语言

#### 4.1.2 图片输入

- **拍照输入**：调用设备摄像头，支持后置摄像头拍照，拍照后可预览并确认/重拍
- **相册选择**：从系统相册选择 1-9 张图片，支持多选
- **图片预处理**：自动压缩至合理尺寸（长边 ≤ 1920px，单张 ≤ 2MB），平衡识别精度与上传速度
- **约束**：单次最多 9 张图片；支持 JPG/PNG/HEIC 格式

#### 4.1.3 音频输入

- **录音输入**：
  - 调用设备麦克风开始录音
  - 录音过程中显示录音时长、波形动画
  - 支持暂停/继续/停止
  - 停止后音频自动保存为临时文件
- **文件选择**：从设备文件系统选择音频文件
- **约束**：单次音频时长上限 30 分钟；支持 m4a/mp3/wav/aac 格式；音频文件大小上限 50MB

#### 4.1.4 输入组合

- 用户可混合输入多种素材，例如：文本 + 图片、音频 + 文本、图片 + 音频 + 文本
- 所有输入素材在"新建笔记"页面统一展示，用户可随时添加/删除素材
- 点击"生成笔记"按钮后，所有素材一并发送至 AI 处理模块

---

### 4.2 AI 处理模块

核心智能模块，负责将原始素材转化为结构化笔记。

#### 4.2.1 OCR 文字识别

- **功能**：对输入图片进行 OCR，提取图片中的文字内容
- **实现方式**：
  - 方案 A（推荐）：调用大模型的多模态能力（如通义千问 VL、智谱 GLM-4V），直接将图片送入模型进行识别
  - 方案 B：使用专用 OCR 服务（如阿里云 OCR）先提取文字，再送入 LLM 整理
- **输出**：每张图片对应一段识别文本，保留原图与文字的映射关系

#### 4.2.2 语音转文字（ASR）

- **功能**：将音频转换为文字
- **实现方式**：
  - 方案 A（推荐）：调用大模型的音频理解能力（如通义千问 Audio），直接发送音频文件
  - 方案 B：使用专用 ASR 服务（如阿里云语音识别）先转写，再送入 LLM 整理
- **输出**：转写文本，保留时间戳信息（用于后续对齐）
- **P1 增强**：录音过程中实时转写，边录边显示文字

#### 4.2.3 LLM 内容整理

- **功能**：将文本、OCR 结果、转写文本等素材综合送入大模型，生成结构化笔记
- **处理流程**：
  1. 素材汇总：将所有输入素材拼接为结构化 prompt
  2. 指令构建：根据笔记类型（通用/会议纪要/课堂笔记等）构建 system prompt
  3. 模型调用：调用用户配置的大模型 API
  4. 结果解析：解析模型返回，提取标题、摘要、要点、正文等结构化字段
- **输出格式**：
  ```json
  {
    "title": "笔记标题",
    "summary": "一句话摘要",
    "keyPoints": ["要点1", "要点2", "要点3"],
    "content": "完整笔记正文（Markdown格式）",
    "tags": ["自动标签1", "自动标签2"]
  }
  ```
- **约束**：
  - 模型调用超时时间 60 秒
  - 支持 streaming（流式输出），用户可看到笔记逐步生成
  - 生成失败时提示错误原因并支持重试

#### 4.2.4 处理流程编排

```
用户点击"生成笔记"
    │
    ├── 有图片？──→ OCR 识别 ──┐
    │                           │
    ├── 有音频？──→ ASR 转写 ──┤
    │                           │
    ├── 有文本？──────────────→ 素材汇总 ──→ LLM 整理 ──→ 结构化笔记
    │
    └── 全部完成，保存并跳转笔记详情页
```

### 4.3 笔记管理模块

负责笔记的存储、展示、编辑、组织与检索。

#### 4.3.1 笔记列表

- 以卡片形式展示笔记，每张卡片显示：标题、摘要（2行截断）、创建时间、分类标签、素材图标（📷📷 / 🎤 / 📝）
- 顶部支持搜索栏（P1）
- 支持按分类筛选（P1）
- 支持排序：按时间倒序（默认）/ 按标题

#### 4.3.2 笔记详情

- **笔记内容区**：展示标题、摘要、要点列表、正文（Markdown 渲染）
- **原始素材区**：展示原始输入素材（图片缩略图、转写文本），可展开/折叠
- **操作栏**：编辑（P1）、分享（P1）、删除（P1）、AI 重新生成（P1）
- **元信息**：创建时间、修改时间、AI 模型、分类标签

#### 4.3.3 笔记编辑（P1）

- 支持富文本编辑：标题、有序/无序列表、加粗、引用
- 编辑后自动保存，支持手动保存
- 编辑不覆盖 AI 原始生成结果（可通过"查看原始版本"对比）

#### 4.3.4 分类管理（P1）

- 支持创建自定义分类/标签
- 每条笔记可分配一个分类 + 多个标签
- AI 生成笔记时自动推荐标签
- 支持在笔记列表按分类/标签筛选

#### 4.3.5 搜索（P1）

- 支持全文搜索（标题 + 正文内容）
- 支持按分类/标签组合筛选
- 搜索结果高亮关键词

#### 4.3.6 分享（P1）

- 分享为纯文本
- 分享为长图（将笔记渲染为图片）
- 分享为 Markdown 文本
- 通过系统分享面板调用其他 App

### 4.4 用户设置模块

负责应用配置与用户偏好。

#### 4.4.1 AI 模型配置（P0）

- **模型提供商选择**：下拉选择支持的大模型（通义千问 / 智谱 GLM / 文心一言）
- **API Key 输入**：用户输入对应提供商的 API Key
- **API Key 加密存储**：API Key 在本地加密存储，不明文展示（输入框显示为密码格式）
- **连通性测试**：输入 API Key 后可点击"测试连接"按钮，验证 Key 是否有效
- **模型参数配置**（高级）：可配置 temperature、max_tokens 等参数（默认值，普通用户无需调整）

#### 4.4.2 偏好设置

- 主题模式：浅色 / 深色 / 跟随系统（P1）
- 默认笔记类型：通用 / 会议纪要 / 课堂笔记（P2）
- 音频录制质量：标准 / 高清（影响文件大小）
- 数据管理：清除缓存、导出全部数据（P2）

## 6. 待确认问题

以下问题需与用户/团队进一步确认：

| 编号 | 问题 | 影响范围 | 建议 |
|------|------|----------|------|
| Q-01 | **AI 多模态能力 vs 专用服务**：OCR 和 ASR 是直接使用大模型的多模态能力（如通义千问 VL/Audio），还是使用专用 OCR/ASR 服务？前者开发简单但依赖模型能力，后者精度更高但需额外集成。 | AI处理模块、架构设计 | 建议 MVP 先用大模型多模态能力快速验证，P1 阶段评估是否引入专用服务提升精度 |
| Q-02 | **数据存储方案**：笔记数据是仅本地存储（SQLite/Realm），还是需要后端服务支持云端同步？MVP 是否需要后端？ | 架构设计、成本 | 建议 MVP 采用纯本地存储（SQLite），P2 迭代增加云端同步 |
| Q-03 | **大模型 API 费用承担方**：用户自配 API Key 则费用由用户承担。是否需要提供官方代理 API（平台统一计费）作为备选？ | 商业模式、用户体验 | 建议 MVP 仅支持用户自配 Key，后续根据市场反馈决定是否提供平台代理 |
| Q-04 | **音频实时转写是否纳入 MVP**：实时转写体验更好但技术复杂度较高（需 WebSocket 流式 ASR），是否纳入 P0？ | MVP 范围、开发周期 | 建议实时转写列为 P1，MVP 先实现录音结束后统一转写 |
| Q-05 | **笔记内容的 Markdown 渲染方案**：AI 生成 Markdown 格式笔记，前端需选择 RN Markdown 渲染库。是否有偏好的库？ | 技术选型 | 建议架构师评估 `react-native-markdown-display` 或 `@ronradtke/react-native-markdown-display` |
| Q-06 | **iOS 编译与测试方案**：当前开发环境为 Linux，无法编译 iOS 包。iOS 端如何测试验证？ | 开发流程、测试 | 建议使用 CI/CD（如 GitHub Actions macOS runner）或远程 Mac 进行 iOS 构建，开发阶段以 Android 为主验证 |

## 7. MVP 范围总结

### 7.1 MVP 包含（P0 + P1）

| 模块 | 功能 |
|------|------|
| 输入 | 文本输入、图片拍照/相册选择、音频录制/文件选择 |
| AI处理 | OCR识别、语音转写、LLM内容整理、多模态融合 |
| 笔记管理 | 列表、详情、保存、编辑、删除、分类、搜索、分享 |
| 设置 | AI模型配置、API Key管理、深色模式 |
| 交互 | AI重新生成、笔记排序 |

### 7.2 MVP 不包含（P2 后续迭代）

- 视频输入与处理
- 笔记模板系统
- 多端云同步
- 多人协作
- 笔记导出（PDF/Word）
- 历史版本管理
- 批量操作

### 7.3 MVP 验收标准

1. 用户可成功配置至少一种国内大模型 API Key 并通过连通性测试
2. 用户可通过文本/图片/音频三种方式输入素材，并成功生成结构化笔记
3. 生成的笔记包含标题、摘要、要点列表、正文，且内容与输入素材相关
4. 笔记列表正确展示所有笔记，可进入详情查看完整内容
5. 笔记可编辑、删除、分类、搜索、分享
6. 支持 Android 端完整功能验证（iOS 提供可编译源码）

## 8. 技术约束说明

| 约束 | 说明 |
|------|------|
| 开发框架 | React Native（跨平台 iOS/Android） |
| AI 模型 | 国内大模型（通义千问/智谱GLM/文心一言），需支持多模态（图片理解、音频理解） |
| API Key | 用户自行配置，本地加密存储，不经过第三方服务器中转 |
| 开发环境 | Linux 环境，无法直接编译 iOS 包，需提供完整可编译源码，iOS 构建通过 CI/CD 或远程 Mac |
| 跨平台兼容 | 需同时兼容 iOS 和 Android，注意摄像头、麦克风、文件系统等原生权限处理 |
| 离线能力 | 笔记数据本地存储，支持离线查看；AI 生成功能需联网 |
